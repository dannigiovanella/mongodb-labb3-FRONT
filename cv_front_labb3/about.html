Om webbplatsen

Detta är en webbapplikation som används för att hantera arbetslivserfarenheter. Projektet består av ett REST API i backend och ett frontend-gränssnitt som kommunicerar med API. Syftet med projektet är att visa hur CRUD-operationer fungerar i praktiken och hur data kan skickas mellan frontend och backend.

Databas och server (uppdaterad)

Applikationen använder MongoDB Atlas som databas istället för en relationsdatabas. Databasen är kopplad till ett Node.js/Express-API som hanterar all kommunikation mellan frontend och databasen inklusive CRUD-operationer.

I stället för tabeller används schema och dokument med Mongoose där struktur och validering av data hanteras direkt i backend, även om validering även hanteras på frontend sidan. API:t är deployat via Render där även databasanslutningen konfigureras med hjälp av miljövariabler från .env.

Funktionalitet

Applikationen gör det möjligt att:

  • Lägga till nya arbetslivserfarenheter
  • Visa sparade poster från databasen
  • Ta bort poster direkt i gränssnittet

Validering av inmatning av data i formulär styrs med tydliga felmeddelande för varje fält för att inte lagra tomma värden.

Slutsats (uppdaterad)

Under arbetet har jag fått en bättre förståelse för hur ett REST API fungerar tillsammans med en NoSQL-databas. Jag har lärt mig hur CRUD-operationer kan implementeras med MongoDB och hur Mongoose används för att strukturera och validera data.

En tydlig skillnad jämfört med att arbeta med SQL är att MongoDB och Mongoose upplevs mer flexibelt och enklare att arbeta med, särskilt genom användningen av scheman och färdiga metoder som create, findByIdAndUpdate och findByIdAndDelete. Personligen upplever jag det som att det blir mindre och mer strukturerad kod vilket gör den mer lättarbetad.

Då mitt arbetssätt varit att uppdatera ett befintligt projekt skapad med SQL och anppassa detta till NOSQL och MongoDB så har det varit klurigt men viktigt att förstå vad som behövs justeras i både BACK och FRONT för att data ska kunna skickas korrekt. Smådetaljer som skillnad på exempelvis hur id används i båda delar är lätt att missa om kommunikationstänket inte finns med från början.

Arbetet har också lärt mig att nekad åtkomst till databasen inte nödvändigtvis behöver ligga i felaktigheter i koden utan hur MongoDB i det här fallet jobbar med vem som kan ha åtkomst, och att det kan varta bra att kolla inställningar nogrannare på databassidan innan man börjar testa att felsöka i koden.

Det exempel felsökning är bara ett av alla som hjälpt mig förstå hur viktigt att strukturera koden tydligt för att kunna identifiera problem vid felsökning.
